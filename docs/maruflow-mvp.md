# MaruFlow MVP 仕様書

## 1. 背景
学校現場において、紙のテスト採点と成績管理は教員の多大な業務負担となっている。
特に「手書きでの丸付け」「点数集計」「システムへの転記」という一連の作業は、自動化の余地が大きい。
将来的な「AIによる完全自動採点」を見据えつつも、まずは「PDF上でデジタル採点を行い、結果を管理・出力する」という基本的なワークフローを確立することが急務である。

## 2. 解決したい課題
- **紙の持ち運び・紛失リスク**: 大量の答案用紙を持ち歩く負担と、紛失のリスクを軽減したい。
- **採点ミスの防止**: 点数の計算間違いや、採点漏れを防ぎたい。
- **集計作業の効率化**: 採点結果を成績管理システム（Excel等）へ手入力する手間を省きたい。
- **データ化の遅れ**: 紙のままでは生徒ごとの学習傾向や弱点をデータとして分析・活用できない。

## 3. MVP (Minimum Viable Product)
最初のMVPでは、技術的ハードル（AI精度やOCR精度）を避け、**「手動でのデジタル採点ワークフローの確立」**に特化する。

- **対象機能**:
  - 空白問題PDFの登録と、採点欄（ゾーン）の手動定義。
  - 答案画像/PDFの読み込みと表示。
  - 画面上での手動採点（丸・バツ・三角の付与、点数入力、コメント入力）。
  - 採点結果の自動集計。
  - 採点済みPDFの生成と出力。

## 4. 非MVP (次フェーズ以降の拡張)
以下の機能はMVPには含めず、将来的な拡張として計画する。

- **AI採点**: GPT-4o Vision等を活用した自動丸付け・自動採点。
- **OCR (手書き文字認識)**: 記述式解答のテキスト化と部分点判定の補助。
- **クラス管理・名簿連携**: 児童生徒の名簿データ（CSV/API）との紐付け。
- **成績分析ダッシュボード**: 問題別の正答率分析や、個人別の成績推移グラフの表示。

## 5. 想定ユーザー
- **小学校・中学校の教員**: 日常的に小テストや単元テストを実施し、手動で採点・集計を行っている教員。

## 6. 画面一覧
MVPモックアップ（`apps/maruflow-mock`）にて実装した画面構成は以下の通り。

1. **ホーム画面**: アプリの概要、ワークフローの全体像、およびMVP範囲の説明を表示する。
2. **PDF登録画面**: 空白問題PDFをアップロードし、テストの基本情報（教科・学年・テスト名）を入力する。
3. **採点欄設定画面**: PDF上に問題ごとの採点ゾーン（座標・サイズ）と配点を定義する。
4. **採点画面**: 答案PDFを表示し、定義したゾーンに対して「丸・バツ・三角」、点数、コメントを手動で入力する。
5. **出力確認画面**: 採点結果のサマリー（平均点等）を確認し、採点済みPDFをダウンロードする。

## 7. 将来拡張へのロードマップ
- **Phase 1 (MVP)**: デジタル採点ワークフローの確立（本仕様）。
- **Phase 2**: 画像認識AI（GPT-4o Vision等）を導入し、客観問題の自動丸付けを実現。
- **Phase 3**: OCR技術を統合し、記述問題のテキスト化と採点補助機能を実装。
- **Phase 4**: 外部の校務支援システムや名簿データとのAPI/CSV連携機能を強化。

## 8. 技術候補 (本番実装時)
- **フロントエンド**: React, TypeScript, Vite, Tailwind CSS (または類似のユーティリティファーストCSS)。
- **PDF処理**: `pdf-lib` や `pdf.js` を用いたブラウザサイドでのPDF描画・生成。
- **バックエンド/BaaS**: Supabase または Firebase (認証・データベース・ストレージ)。※将来のデータ永続化・共有用。
- **AI/OCR (将来)**: OpenAI API (GPT-4o), Google Cloud Vision API 等。

## 9. 注意点
- **個人情報の取り扱い**: 実際の児童生徒の氏名や答案（筆跡含む）は機微な個人情報であるため、本番実装時はセキュアなデータ保管（暗号化、アクセス制御）が必須となる。MVPモックでは架空データのみを使用する。
- **UI/UXのレスポンシブ対応**: 採点作業はタブレット端末（iPad等）で行われる可能性が高いため、タッチ操作に適したUI設計（ボタンサイズ、スワイプ操作等）を考慮する必要がある。
